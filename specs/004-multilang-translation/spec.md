# Feature Specification: 多语言翻译功能增强

**Feature Branch**: `004-multilang-translation`  
**Created**: 2025-11-17  
**Status**: Draft  
**Input**: 多语言翻译功能增强：支持所有8种语言的文档上传翻译、新闻焦点和Latest编辑自动翻译、以及网站各模块的完整多语言支持

## 概述

当前网站支持8种语言（简体中文、繁体中文、英语、日语、西班牙语、法语、阿拉伯语、印地语），但翻译功能存在以下限制：

1. **文档上传翻译**：目前只支持中文自动翻译成英文
2. **新闻编辑功能**：焦点新闻（Focus Point）和特色文章（Featured Articles）的编辑界面只支持中英文双语
3. **网站模块翻译**：部分模块不能切换到所有8种语言
4. **部署便利性**：需要确保翻译功能改进后能方便地重新部署到AWS服务器

本规范旨在将翻译功能扩展到支持所有8种语言，并保持现有的工作原理（DeepSeek AI翻译 + 缓存机制）。

## User Scenarios & Testing

### User Story 1 - 文档上传多语言翻译 (Priority: P1)

作为内容管理员，当我上传文档时，我希望能够选择将文档翻译成任意支持的语言（不仅仅是英文），这样我可以一次性生成多语言版本的文档内容。

**Why this priority**: 这是最核心的功能改进，直接影响内容创建效率。文档上传是内容管理的主要入口，支持多语言翻译可以大幅提升工作效率。

**Independent Test**: 可以通过上传一个中文Markdown文档，选择目标语言（如日语、法语等），验证翻译结果是否正确保存到数据库，并且可以在前端正确显示。

**Acceptance Scenarios**:

1. **Given** 管理员在文档上传界面，**When** 上传一个中文文档并选择目标语言为"日语"，**Then** 系统自动将文档内容翻译成日语并保存
2. **Given** 管理员在文档上传界面，**When** 上传一个英文文档并选择目标语言为"简体中文"，**Then** 系统自动检测源语言为英文，并翻译成简体中文
3. **Given** 管理员在文档上传界面，**When** 选择多个目标语言（如日语、法语、西班牙语），**Then** 系统并发翻译成所有选择的语言
4. **Given** 文档包含Markdown图片，**When** 翻译完成后，**Then** 图片链接保持不变，正确嵌入到翻译后的内容中
5. **Given** 翻译过程中出现错误，**When** 某个语言翻译失败，**Then** 系统显示具体错误信息，但不影响其他语言的翻译

---

### User Story 2 - 新闻编辑多语言支持 (Priority: P1)

作为内容管理员，当我编辑焦点新闻（Focus Point）或特色文章（Featured Articles）时，我希望能够为所有8种语言提供内容，并且可以使用自动翻译功能快速生成多语言版本。

**Why this priority**: 新闻是网站的核心内容，焦点新闻和特色文章是用户最先看到的内容。支持多语言编辑可以确保所有语言用户都能获得完整的新闻体验。

**Independent Test**: 可以通过编辑一条焦点新闻，填写中文内容后点击"自动翻译"按钮，验证是否生成了所有7种其他语言的翻译版本，并且可以在前端切换语言查看。

**Acceptance Scenarios**:

1. **Given** 管理员在编辑焦点新闻，**When** 填写中文标题和内容后点击"翻译到所有语言"按钮，**Then** 系统自动生成其他7种语言的翻译
2. **Given** 管理员在编辑特色文章，**When** 只填写英文内容，**Then** 可以选择翻译到其他语言（包括中文、日语等）
3. **Given** 焦点新闻编辑表单，**When** 显示时，**Then** 提供8种语言的输入框（可折叠/展开），每种语言旁边有"翻译"按钮
4. **Given** 管理员编辑了某个语言的内容，**When** 点击该语言旁边的"翻译"按钮，**Then** 可以选择翻译到哪些目标语言
5. **Given** 翻译完成后，**When** 保存新闻，**Then** 所有语言版本都保存到localStorage（或数据库）

---

### User Story 3 - Latest新闻多语言支持 (Priority: P2)

作为内容管理员，当我添加或编辑Latest实时新闻时，我希望能够为所有8种语言提供内容，并使用自动翻译功能。

**Why this priority**: Latest新闻是实时动态内容，虽然重要性略低于焦点新闻，但也需要多语言支持以服务全球用户。

**Independent Test**: 可以通过添加一条Latest新闻，填写中文内容后使用自动翻译，验证是否生成了所有语言版本，并且在前端切换语言时正确显示。

**Acceptance Scenarios**:

1. **Given** 管理员在添加Latest新闻，**When** 填写中文标题和摘要，**Then** 可以点击"翻译到所有语言"生成其他语言版本
2. **Given** Latest新闻编辑表单，**When** 显示时，**Then** 提供8种语言的输入框，支持单独编辑或批量翻译
3. **Given** 用户在前端查看Latest新闻，**When** 切换语言，**Then** 显示对应语言的新闻内容
4. **Given** 某条Latest新闻缺少某种语言的翻译，**When** 用户切换到该语言，**Then** 显示默认语言（中文或英文）的内容

---

### User Story 4 - 网站模块完整多语言支持 (Priority: P2)

作为网站访问者，当我切换网站语言时，我希望所有模块（导航、首页、业务页面、新闻页面、咨询页面、联系页面）都能完整显示选择的语言，没有遗漏或混合语言的情况。

**Why this priority**: 这是用户体验的完整性保证。虽然不影响核心功能，但对于非中英文用户来说非常重要。

**Independent Test**: 可以通过切换到每种语言（日语、西班牙语、法语、阿拉伯语、印地语），逐个检查每个页面的每个模块，验证是否所有文本都正确翻译。

**Acceptance Scenarios**:

1. **Given** 用户在首页，**When** 切换到日语，**Then** 导航栏、Hero区域、统计数据、特色业务等所有模块都显示日语
2. **Given** 用户在业务页面，**When** 切换到法语，**Then** 所有业务类别、描述、标题都显示法语
3. **Given** 用户在新闻页面，**When** 切换到西班牙语，**Then** 新闻分类、文章标题、摘要都显示西班牙语
4. **Given** 用户在咨询页面，**When** 切换到阿拉伯语，**Then** AI聊天界面、预约表单、服务介绍都显示阿拉伯语（注意RTL布局）
5. **Given** 用户在联系页面，**When** 切换到印地语，**Then** 表单标签、按钮、提示信息都显示印地语
6. **Given** 发现某个模块缺少翻译，**When** 切换语言时，**Then** 该模块显示英文作为后备语言

---

### User Story 5 - AWS部署便利性 (Priority: P3)

作为开发者，当我完成翻译功能改进后，我希望能够方便地将更新部署到AWS服务器，包括前端（S3 + CloudFront）和后端（EC2），并且不影响现有数据和服务。

**Why this priority**: 这是运维需求，确保功能改进能够顺利上线。虽然优先级较低，但对于生产环境部署至关重要。

**Independent Test**: 可以通过运行部署脚本，验证前端和后端是否成功更新，服务是否正常运行，现有数据是否完整。

**Acceptance Scenarios**:

1. **Given** 代码已提交到main分支，**When** 触发GitHub Actions工作流，**Then** 前端自动构建并部署到S3，CloudFront缓存自动刷新
2. **Given** 后端代码已更新，**When** 运行部署脚本，**Then** EC2服务器拉取最新代码，安装依赖，重启服务
3. **Given** 数据库需要更新（如添加新的语言字段），**When** 运行数据库迁移，**Then** 现有数据保持完整，新字段正确添加
4. **Given** 部署完成后，**When** 访问生产环境网站，**Then** 所有翻译功能正常工作，性能无明显下降
5. **Given** 部署过程中出现错误，**When** 检查日志，**Then** 可以快速定位问题并回滚到上一个稳定版本

---

## Technical Constraints

1. **保持现有翻译机制**：继续使用DeepSeek AI进行翻译，保持缓存机制（30天过期）
2. **数据库兼容性**：确保新的多语言字段与现有PostgreSQL数据库兼容
3. **性能要求**：批量翻译时使用并发处理，单次翻译不超过5秒
4. **错误处理**：翻译失败时提供清晰的错误信息，不影响其他功能
5. **向后兼容**：现有的中英文双语内容应该继续正常工作
6. **部署要求**：前端部署到AWS S3 + CloudFront，后端部署到AWS EC2，使用GitHub Actions自动化部署

## Out of Scope

1. 不改变现有的翻译API提供商（继续使用DeepSeek）
2. 不实现人工翻译审核工作流
3. 不实现翻译质量评分系统
4. 不实现翻译记忆库（Translation Memory）
5. 不改变现有的数据库架构（只添加新字段，不重构表结构）

## Success Metrics

1. **功能完整性**：所有8种语言都能在文档上传、新闻编辑、网站浏览中正常使用
2. **翻译准确性**：抽样检查翻译质量，确保基本语义正确
3. **性能指标**：单个文档翻译到7种语言的总时间不超过30秒
4. **用户体验**：语言切换流畅，无明显延迟或错误
5. **部署成功率**：部署脚本成功率达到95%以上

## Dependencies

1. DeepSeek API（需要支持所有8种语言的翻译）
2. PostgreSQL数据库（需要添加新的语言字段）
3. AWS S3 + CloudFront（前端部署）
4. AWS EC2（后端部署）
5. GitHub Actions（自动化部署）

## Risks & Mitigations

1. **风险**：DeepSeek API可能不支持某些语言对的翻译
   - **缓解**：提前测试所有语言对，如有不支持的，考虑使用英文作为中间语言（如中文→英文→阿拉伯语）

2. **风险**：批量翻译可能导致API速率限制
   - **缓解**：实现请求队列和重试机制，添加速率限制提示

3. **风险**：数据库迁移可能导致数据丢失
   - **缓解**：部署前备份数据库，使用Alembic进行版本化迁移

4. **风险**：前端打包体积增大（8种语言的翻译文件）
   - **缓解**：使用代码分割，按需加载语言包

5. **风险**：阿拉伯语RTL布局可能导致UI错乱
   - **缓解**：使用CSS的`dir`属性和Tailwind的RTL支持，充分测试

## Notes

- 当前支持的8种语言：zh-CN（简体中文）、zh-TW（繁体中文）、en（英语）、ja（日语）、es（西班牙语）、fr（法语）、ar（阿拉伯语）、hi（印地语）
- 翻译缓存使用SHA-256哈希，存储在PostgreSQL的`translation_cache`表中
- 文档上传功能位于`backend/app/routers/documents.py`
- 新闻编辑功能位于`src/components/NewsPage.tsx`（焦点新闻和特色文章）
- Latest新闻编辑功能也在`src/components/NewsPage.tsx`
- 多语言翻译文件位于`src/i18n/translations.ts`
- 部署配置位于`.github/workflows/deploy-frontend.yml`和`.github/workflows/deploy-backend.yml`
